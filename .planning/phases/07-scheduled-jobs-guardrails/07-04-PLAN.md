<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| workflow → second git remote | backup push crosses to a new trust surface (BACKUP_REMOTE_URL credential) |
| workflow → GitHub API | actions-minutes step calls gh api with GITHUB_TOKEN |
| workflow → vault log.md / dashboard.md | nightly automated writes to the human-readable audit surface |
| dispatch payload → workflow | repository_dispatch client_payload is untrusted (comes from the capture Worker) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04-01 | Spoofing | backup remote credential | mitigate | BACKUP_REMOTE_URL is a repo secret (masked in logs); scoped to a backup mirror repo only (contents:write on that one repo), not the production vault — Plan 05 checkpoint enforces scope |
| T-07-04-02 | Information disclosure | BACKUP_REMOTE_URL in logs | mitigate | reference via `${{ secrets.BACKUP_REMOTE_URL }}` so GitHub masks it; never echo the URL; backup push step uses it directly in git push |
| T-07-04-03 | Tampering | workflow injection via dispatch payload | mitigate | ingest.yml does NOT interpolate client_payload into shell; all steps use fixed commands with $GITHUB_WORKSPACE / secrets only — no `${{ github.event.client_payload.* }}` in run: blocks |
| T-07-04-04 | DoS / cost | Actions minutes runaway | mitigate | timeout-minutes: 10 (existing) + 80% alert (GUARD-06) + per-run token/cost caps in batch_ingest; continue-on-error on backfill/lint prevents retry storms |
| T-07-04-05 | Repudiation | backup push failure | accept | continue-on-error: true on backup step means an outage is visible in the run's step status (yellow) but does not abort ingest; operator monitors run history |
| T-07-04-06 | Elevation of privilege | GITHUB_TOKEN scope | mitigate | job permissions stay contents: read; backup push uses BACKUP_REMOTE_URL's embedded credential, not the GITHUB_TOKEN — no permissions: write escalation |
</threat_model>
